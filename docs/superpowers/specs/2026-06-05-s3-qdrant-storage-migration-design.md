# S3 Qdrant Storage Migration Design

## Goal

Move the current local source storage and persistent vector storage to a cloud
backed design using AWS S3 and Qdrant, while preserving the existing public API
and shared contracts.

The migration should cover:

- Raw uploaded PDF files.
- URL and text source records.
- Parsed Markdown and debug artifacts.
- Chunk manifests and source listings.
- Persistent vector search storage.

The design should avoid adding Postgres for v1 and should optimize for a
prototype or free-first deployment posture.

## Current State

The repository currently uses `LocalPdfEvidenceProvider` as the API-facing
source provider for PDF, URL, and text ingestion.

Current persistence behavior:

- Raw PDF files are saved under `storage/local_pdf/files/`.
- Parsed Markdown is saved under `storage/local_pdf/parsed/`.
- Debug artifacts are saved under `storage/local_pdf/debug/` and
  `storage/local_pdf/artifacts/`.
- Chunks are saved as per-document JSONL files under
  `storage/local_pdf/chunks/`.
- Optional source metadata and chunks can also be stored in Postgres through
  `PostgresLocalSourceStore`.
- Dense embeddings are only persisted when `DENSE_VECTOR_STORE=pgvector`; the
  default `turbovec` path is local and in-process.

Current retrieval behavior:

- `LocalPdfEvidenceProvider.retrieve()` loads chunks for the requested
  documents.
- `Store(chunks)` rebuilds BM25 in memory on each retrieval request.
- Dense retrieval uses the configured vector store.
- Fusion and generation consume shared `SearchResult` objects.

This request-time BM25 rebuild is acceptable for small local runs but adds
latency and repeated work in a cloud deployment.

## Target Architecture

Keep `LocalPdfEvidenceProvider` as the API-facing provider and add cloud-backed
adapters behind it instead of introducing a new provider name.

Cloud storage split:

- AWS S3 is the durable storage system for source files and ingestion
  artifacts.
- Qdrant is the persistent retrieval index for cloud mode.

S3 should store:

- Raw uploaded PDFs.
- URL source records and text source records.
- Parsed Markdown.
- Debug artifacts generated by PDF and URL ingestion.
- Per-document chunk manifests.
- Per-document source manifest metadata used by `/sources`, `/debug`, and
  delete operations.

Qdrant should store one point per chunk, with:

- Dense vector.
- Sparse vector for lexical or hybrid retrieval.
- Chunk text.
- Payload fields needed for filtering and result reconstruction.

Recommended payload fields:

- `document_id`
- `chunk_id`
- `storage_chunk_id`
- `source_type`
- `source`
- `url`
- `page`
- `section`
- `text`
- `metadata`

This keeps Qdrant on the hot path for retrieval and keeps S3 as the durable
audit and artifact layer.

## Storage Model

Use deterministic S3 document prefixes so one document can be listed, debugged,
or deleted without scanning the full bucket.

Suggested shape:

```text
<prefix>/<document_id>/
  manifest.json
  raw/source.pdf
  raw/source.txt
  parsed/document.md
  chunks/chunks.jsonl
  debug/...
  artifacts/...
```

`manifest.json` should contain:

- `document_id`
- `dataset_id`
- `provider`
- `name`
- `source_type`
- `source`
- `total_chunks`
- `created_at`
- `updated_at`
- S3 object keys for raw content, parsed Markdown, chunk manifest, and debug
  assets
- Lightweight metadata needed by the list and debug endpoints

Chunk manifests should preserve the current shared `Chunk` representation so
debugging and contract compatibility remain straightforward.

Deduplication is out of scope for v1. The design may still use deterministic
IDs for chunk upserts and deletes, but it should not try to collapse duplicate
documents or content.

## Retrieval Design

Cloud mode should not rebuild BM25 from S3 chunk text on each request.

Instead:

- Qdrant stores both dense and sparse vectors for each chunk.
- Retrieval uses Qdrant hybrid search with document filters when
  `document_ids` are provided.
- Qdrant returns chunk text and metadata payload sufficient to rebuild
  `SearchResult` objects directly.
- Existing fusion and generation boundaries stay intact, but cloud mode may
  skip the local BM25 builder because Qdrant already provides the lexical and
  dense candidate set.

Local compatibility remains:

- Existing `jsonl`, `postgres`, `turbovec`, and `pgvector` paths stay available.
- Existing local BM25 rebuild behavior remains the fallback for local mode.

This gives the project a lean cloud path without introducing OpenSearch or a
metadata database.

## Ingestion Flow

For PDF upload:

1. Accept the upload through the existing `/sources/upload` endpoint.
2. Parse and chunk using the current local ingestion pipeline.
3. Write raw file, parsed Markdown, chunk manifest, and debug artifacts to S3.
4. Upsert chunk points into Qdrant with vectors, text, and metadata payload.
5. Save the source manifest to S3.

For URL upload:

1. Fetch and parse the URL through the existing URL ingestion path.
2. Save source metadata, parsed Markdown, debug artifacts, and chunk manifest to
   S3.
3. Upsert URL-derived chunks into Qdrant.
4. Save the source manifest to S3.

For text upload:

1. Chunk the input text through the current text ingestion path.
2. Save the source record, parsed text or Markdown, and chunk manifest to S3.
3. Upsert text-derived chunks into Qdrant.
4. Save the source manifest to S3.

The current trace payload shape should remain compatible, but cloud mode should
report S3 and Qdrant as the write targets instead of JSONL or pgvector.

## API And Provider Behavior

Keep the current API request and response models unchanged.

Behavior by endpoint:

- `/sources/upload`, `/sources/url`, `/sources/text`:
  continue to return the existing `SourceUploadResponse`.
- `/sources`:
  list source manifests from S3 and hydrate summary metadata from
  `manifest.json`.
- `/sources/{document_id}/chunks`:
  read chunk manifests from S3 when needed, unless Qdrant payload retrieval is
  sufficient for the requested view.
- `/sources/{document_id}/debug`:
  read manifest, parsed Markdown, and chunk manifest from S3.
- `/sources/{document_id}/raw`:
  stream the underlying S3 object through the API so the endpoint contract does
  not change.
- `/sources/{document_id}` delete:
  remove the S3 document prefix and delete Qdrant points filtered by
  `document_id`.
- `/sources` delete all:
  delete all stored source prefixes under the configured S3 prefix and clear the
  matching Qdrant collection or delete matching points.

Reusing `EVIDENCE_PROVIDER=local_pdf` avoids front-end and API contract churn.

## Configuration

Keep the current provider selection model and extend environment-driven storage
selection.

Add or document:

- `LOCAL_SOURCE_STORE=s3`
- `AWS_REGION`
- `AWS_S3_BUCKET`
- `AWS_S3_PREFIX`
- `DENSE_VECTOR_STORE=qdrant`
- `QDRANT_URL`
- `QDRANT_API_KEY`
- `QDRANT_COLLECTION`

Existing options remain valid:

- `LOCAL_SOURCE_STORE=jsonl`
- `LOCAL_SOURCE_STORE=postgres`
- `DENSE_VECTOR_STORE=turbovec`
- `DENSE_VECTOR_STORE=pgvector`

Cloud mode should fail fast when required AWS or Qdrant configuration is
missing.

## Error Handling

Expected failure behavior:

- Missing S3 configuration: raise a clear configuration error before upload or
  list operations.
- Missing Qdrant configuration: raise a clear configuration error before
  indexing or retrieval.
- S3 upload failure after local parse:
  fail the request and do not report the document as available.
- Qdrant upsert failure after S3 writes:
  fail the request and attempt best-effort cleanup of the S3 document prefix.
- Delete failure:
  report the error clearly and avoid silently leaving half-deleted state where
  possible.

For v1, rollback and cleanup should be best-effort rather than fully
transactional. The design should not assume distributed transactions across S3
and Qdrant.

## Testing

Add deterministic tests with mocked clients only.

Storage tests:

- S3 source store writes and reads source manifests correctly.
- S3 source store returns stored chunks and source metadata correctly.
- S3 source store streams raw content and parsed Markdown correctly.
- S3 delete operations remove the expected document prefix.

Qdrant tests:

- Qdrant point upsert uses deterministic IDs and expected payload fields.
- Qdrant retrieval honors `document_id` filters.
- Qdrant retrieval reconstructs `SearchResult` and `Chunk` payloads correctly.
- Qdrant delete removes points for one `document_id`.

Provider and API regression tests:

- PDF upload works in S3 + Qdrant mode.
- URL upload works in S3 + Qdrant mode.
- Text upload works in S3 + Qdrant mode.
- `/sources` lists cloud-backed sources.
- `/sources/{document_id}/debug` returns cloud-backed debug data.
- `/sources/{document_id}/raw` streams the raw S3 object.
- Retrieval returns hybrid results with preserved metadata.
- Delete endpoints remove both S3 and Qdrant state.

Tests must not require:

- AWS credentials
- Qdrant network access
- External vector databases
- Live upload or parser services beyond current mocked local behavior

## Documentation

Update:

- `.env.example` with S3 and Qdrant configuration.
- `README.md` with supported storage and vector backends.
- Relevant local provider or generation docs to explain cloud-backed source
  storage and retrieval behavior.

Document that:

- S3 is the durable storage layer for source artifacts.
- Qdrant is the cloud retrieval index in cloud mode.
- Local JSONL and turbovec remain valid for local-only development.

## Out Of Scope For V1

- Postgres metadata storage.
- OpenSearch or another dedicated sparse search service.
- Automatic deduplication or content reuse.
- Strong transactional rollback across S3 and Qdrant.
- Production backup, retention, and lifecycle policy automation.
- Switching endpoint contracts to presigned URLs.
